## 1. 系統定位

**名稱** GPS Issue & Management Center
**性質** OSINT 驅動的議題偵測與案件管理平台
**服務對象** TSMC GPS 10 人核心團隊
**目的** 為全球廠區安全相關議題提供早期偵測、語用判讀、案件化追蹤與內部協作

**監測對象(待 Phase 0 與 GPS 同事 align)**

1. 地緣政治與外交事件(影響廠區所在地)
2. 實體安全事件(抗議、衝突、犯罪)
3. 供應鏈中斷訊號
4. 監管與制裁變化
5. 針對 TSMC 的敵意論述或組織化動員

**不做的事** 不處理 TSMC 內部資料;不取代既有 GPS 工具;不對外提供 API 或外售服務;不做多模態與社會網絡分析。

## 2. 核心命題

把語用學判讀框架嵌入 Claude 的 prompt 層,讓全球公開資料源在進入 GPS 團隊視線前先經過**威脅升級語言、行為者具體性、地理具體性、時間急迫性**四維度的結構化判讀,並以 Issue 為單位管理生命週期。

---

## 3. 設計原則

1. **評測先於建置** — Phase 0 在寫第一行採集程式前完成 rubric 與評測 harness
2. **單一 Claude 模型優先** — Phase 1-2 全 Sonnet,Phase 3 視成本決定是否引入 Haiku triage
3. **HITL 是常態** — 對外發出的判讀與報告皆需團隊覆核閘門
4. **可攜性高於開發便利** — Phase 1-2 在外部運作,但所有技術選型須能在 Phase 3 平移至 TSMC 內部;**寧可多花一週寫得 portable,不要 Phase 3 重寫**
5. **每週可維運** — 單人開發限制下,所有設計需通過「每週 2 小時維運」測試
6. **採集與判讀解耦** — 任何外部 API 呼叫都假設會失敗或延遲,管線需具備可重試性
7. **Documentation-as-Infrastructure(v0.4 新增)** — 因 bus factor = 1 的結構性風險,所有架構決策、運維程序、評測結果皆需具備「Rockie 不在場仍可被理解與接手」的文件品質;ADR、README、runbook 不是可選項

## 6. 系統架構

### 6.1 整體分層

**採集層 → Queue → 判讀層 → Correlation → 儲存層 → 管理層 → 交付層**

(v0.4 在 v0.3 五層架構之間新增 Correlation 一步,見 §6.7)

### 6.2 採集與判讀解耦 + 並發控制(v0.4 更新)

採集與判讀以 Postgres `ingest_queue` 為界,兩個獨立 Cron 各司其職。

`ingest_queue` 最小欄位:`id`, `source_url`, `source_type`, `raw_content`, `status (pending/processing/done/failed)`, `retry_count`, `created_at`, `processed_at`.

**並發控制(v0.4 新增)** — Inference worker 取任務時必須使用 `FOR UPDATE SKIP LOCKED`:

```sql
SELECT * FROM ingest_queue
WHERE status = 'pending'
ORDER BY created_at ASC
LIMIT 10
FOR UPDATE SKIP LOCKED;
```

理由:Phase 1 末若 Inference Cron 重疊執行(前次未完成新批啟動),沒有 SKIP LOCKED 會雙重處理同一筆。這是 production-grade Postgres queue 的標準作法,零工程成本但失去它必踩坑。

### 6.3 LLM Gateway 抽象(v0.4 更新)

**目的** Phase 3 若 TSMC IT 要求改走 AWS Bedrock 或 GCP Vertex,只改一個函數。

**做法** 所有 Claude 呼叫**僅透過單一函數**,且函數命名**不綁定 provider**:

```typescript
// lib/llm/gateway.ts
async function runTriageInference(
  prompt: string,
  options?: { model?: string; maxTokens?: number }
): Promise<TriageResult>
```

**v0.4 改名理由** v0.3 用 `runClaudeJudgment` 把 provider 名稱寫進 domain vocabulary 是錯的。`runTriageInference` 是 domain action,實作可換。

**紀律** Anthropic SDK 不允許散佈在多個檔案,只能出現在 `lib/llm/gateway.ts`。其他檔案一律 import 這個函數。

**不要做的事** 不要 Phase 1 就建支援三家廠商的完整抽象層、不要引入 LangChain 或類似框架、**不要把 gateway 升級為 "AI Policy Enforcement Layer"**——保持為 30-60 行的單純函數。

### 6.4 Supabase 使用紀律(不變)

**僅取 Postgres 連線字串,放棄 Supabase 特有功能。** Auth、RLS、Realtime、Edge Functions、Storage、`@supabase/supabase-js` SDK 一律不使用。替代方案見 ADR-002。

### 6.5 各層技術選型

| 層 | 選型 | 備註 |
|---|---|---|
| 採集 | Vercel Cron + 自寫 connector(走 §6.6 interface) | — |
| Queue | Postgres `ingest_queue` 表 + SKIP LOCKED | 不引入 Redis / SQS |
| 判讀 | Anthropic SDK(經 LLM Gateway 抽象)+ Sonnet 模型 | 結構化 JSON 輸出 + tool use |
| Correlation | Postgres + fingerprint heuristic(見 §6.7) | 不引入 embedding |
| ORM | Prisma(ADR-006) | — |
| 認證 | NextAuth.js(Auth.js v5+) | Phase 1 簡單 email/password,Phase 3 整合 TSMC SSO |
| 前端 | Next.js 14+ App Router | shadcn/ui 元件庫 |
| 警示 | Email(Resend)、Phase 2 加 Teams 或 LINE | — |
| 監控 | Vercel Analytics + §6.9 自寫 metrics 表 | Phase 1 末上線 |

### 6.6 Source Connector Interface Contract(v0.4 新增)

```typescript
interface SourceConnector {
  readonly id: string;
  readonly displayName: string;
  readonly supportedLanguages: string[];
  fetch(params: FetchParams): Promise<RawArticle[]>;
  parseStatus(): Promise<SourceHealth>;
}
```

Phase 1 工程節奏:Week 3 NewsAPI,Week 4 Google News RSS,Week 5 定義 interface refactor 前兩個 connector,Week 6+ 之後皆實作此 interface。

### 6.7 Issue Correlation Strategy(v0.4 新增)

Fingerprint heuristic,**不用 embedding**:Geo + Actor + Event window(±3 天)+ Keyword cluster。

**Merge 規則** 三項皆重疊 → suggest merge;Geo + Actor 重疊但 window 不重疊 → suggest link;其他 → new issue。

**重要紀律** **永不自動合併**,只 suggest;Phase 2 末若 hit-rate < 70% 再評估 embedding 升級。

### 6.8 Authorization Matrix(v0.4 新增)

| Role | Resource | Action | Scope |
|---|---|---|---|
| viewer | issues | read | all |
| analyst | issues | read, update_status, comment, hitl_correct | all |
| analyst | issues | reassign | self_owned |
| admin | issues | all | all |
| admin | sources | manage | all |
| admin | prompts | manage | all |
| admin | users | manage | all |

實作位置:Next.js Server Actions 層,所有 mutation 前 `requireRole(user, action, resource)`。

### 6.9 Observability(v0.4 新增,4 項關鍵指標)

| 指標 | 頻率 | 用途 |
|---|---|---|
| `queue_backlog_size` | 每 5 分鐘 | 偵測判讀跟不上採集 |
| `ingestion_success_rate` | 每小時 | 偵測 source 失效 |
| `inference_parse_failure_rate` | 每 5 分鐘 | 偵測 prompt 退化或 schema 漂移 |
| `monthly_token_cost_usd` | 每日 | 對接預算熔斷(§9.2) |

**明確不做的:** retry distribution、failed source distribution、hallucinated schema rate、latency p99。

## 7. Issue 生命週期與資料模型

```
Detected → Triaged → Assigned → Investigating → Resolved / Monitoring
```

Issue 核心欄位包含 `category` (5 類), `geo` (jsonb), `primary_actors`, `urgency_score` (1-5), `triage_summary`, `confidence_score`, `status`, `assigned_to`, `metadata` (jsonb 預留), `prompt_version`, `model_version`, `rubric_version`.

`issue_history` 含 `hitl_correction_reason` (enum: `false_positive` / `wrong_category` / `missed_urgency` / `wrong_geo` / `wrong_actors` / `other`) 與 `hitl_correction_note`。

**重要紀律** HITL 修正時介面必須**強制選原因分類**。Phase 2 後分析 prompt 退化原因,這個欄位是金礦。

### 9.2 預算熔斷機制(Phase 1 末啟用)

每日 09:00 cron 拉昨日 token 用量;若昨日成本 > 「日預算 × 1.5」,自動將 Ingestion Cron 暫停旗標設為 `true`;Inference Cron 不停;Email Rockie 告警,**手動恢復**。

日預算公式:月度預算 × 1.1 ÷ 30。

## 14. 治理紀律專章(v0.4 新增)

### 14.1 SLO 定義

| 指標 | 目標 |
|---|---|
| 採集新鮮度 | 文章發布後 6 小時內進入 Queue |
| 判讀完成度 | 進 Queue 後 24 小時內判讀完成 |
| 月度可用率 | > 99% |
| HITL 修正率 | < 15%(Phase 2 末)/ < 10%(Phase 3) |

### 14.2 Data Retention Policy

詳見 `docs/data-retention.md`。

### 14.3 Prompt Injection Hardening(5 條政策)

1. System prompt 硬邊界
2. 外部內容 delimiter 標記
3. Tool call 白名單(僅 `store_judgment`)
4. Output schema 嚴格驗證
5. Unsafe instruction 棄置政策

詳見 `docs/prompt-injection-policy.md`。

### 14.4 LLM Model Version Pinning Policy

Production 鎖定 specific model identifier;不使用 `*-latest`;升版前以新版跑既有 100 筆 ground truth,classification agreement > 95% 才升;升版必撰 ADR。詳見 ADR-007。

### 14.5 Bus Factor = 1 與 Documentation-as-Infrastructure

承認 bus factor = 1 是結構性風險,工程無法解。降低衝擊面紀律:

1. README 品質標準(假設 Rockie 不在,新人 30 分鐘內可啟動)
2. ADR 強制
3. Runbook 漸進累積
4. Prompt 完整文件化
5. Phase 3 移交準備

詳見 `README.md`, `docs/adr/`, `docs/runbook.md`。
